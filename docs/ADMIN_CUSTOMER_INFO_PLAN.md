# /admin 고객 및 주문 정보 표시 설계 문서

- 작성 기준: 2026-07-06
- 대상 화면: `/admin` 오늘 처리, 파일 찾기, 파일 상세 화면
- 문서 성격: 구현 전 설계 문서

## 1. 문서 목적

이 문서는 `/admin`에서 파일을 확인할 때 해당 파일이 어떤 고객 및 주문과 연결되어 있는지 더 쉽게 확인하기 위한 표시 기준을 정리합니다.

현재 `/admin`은 `file_id`와 Cafe24 주문번호를 기준으로 파일을 찾고 처리할 수 있습니다. 다만 실무에서는 “누가 올린 파일인지”, “어느 주문의 파일인지”, “상품명과 옵션이 무엇인지”, “연락이 필요한 고객인지”를 같이 봐야 하는 경우가 있습니다.

이번 설계의 목적은 다음과 같습니다.

- 관리자가 파일 처리 시 주문자와 주문 정보를 빠르게 확인할 수 있게 합니다.
- `file_id`와 주문번호만으로 부족한 운영 정보를 보완합니다.
- 주문자명, 연락처, 이메일, 배송지, 상품명, 옵션 등의 표시 기준을 정리합니다.
- 개인정보는 필요한 화면에만 최소한으로 노출되도록 설계합니다.
- 디자이너 또는 파일 검수 담당자에게 필요한 정보와 관리자에게만 필요한 정보를 구분합니다.

이번 문서는 설계 문서이며, 앱 코드, API, DB, Webhook 로직, Cafe24 운영 스킨은 변경하지 않았습니다.

## 2. 현재 /admin 정보 구조 요약

현재 코드 기준으로 `/admin`에서 확인 가능한 주요 정보는 다음과 같습니다.

| 정보 | 현재 확인 위치 | 비고 |
|---|---|---|
| `file_id` | 오늘 처리, 파일 찾기, 파일 상세 | Supabase `files.id` |
| 파일명 | 오늘 처리, 파일 찾기, 파일 상세 | `files.original_filename` |
| Cafe24 주문번호 | 오늘 처리, 파일 찾기, 파일 상세 | `files.order_id` |
| 파일 상태 | 오늘 처리, 파일 찾기, 파일 상세 | `uploaded_pending`, `reviewing` 등 |
| 업로드일 | 오늘 처리, 파일 상세 | `files.created_at` |
| 다운로드 | 오늘 처리, 파일 상세 | `/api/files/download?file_id=...` |
| 주문번호 연결 이력 | 파일 상세 | `file_order_link_logs` |
| 상태 변경 이력 | 파일 상세 | `file_status_change_logs` |

| Cafe24 주문 조회 테스트 | 파일 찾기 탭 | Cafe24 Admin API 호출 |
| Webhook 처리 상태 | 로그 확인 탭 | `cafe24_webhook_events.processed_status` |

현재 `files` 테이블에는 주문자명, 전화번호, 이메일, 배송지 주소, 상품명, 상품 옵션을 저장하는 전용 컬럼이 없습니다. `files`에는 파일 메타데이터와 주문번호 연결 정보가 중심으로 저장되어 있습니다.

## 3. 현재 Cafe24 주문 조회 코드 기준

현재 Cafe24 주문 조회 관련 코드는 `src/lib/cafe24/order-lookup.ts`에 있습니다.

현재 요약 타입 `Cafe24OrderLookupSummary`에서 정리하는 정보는 다음과 같습니다.

| 구분 | 현재 코드에서 요약하는 값 |
|---|---|
| 주문 기본 | `orderId`, `orderNo`, `orderedAt`, `orderStatus` |
| 품목 | `productName`, `productNo`, `variantCode` |
| 옵션 | `optionText`, `additionalOptionText` |
| 업로드 파일 ID | `uploadFileIds`, `uploadFileIdSources` |
| 응답 구조 확인 | top-level keys, item count, item lookup status |

현재 코드가 직접 요약하지 않는 정보는 다음과 같습니다.

- 주문자명
- 주문자 이메일
- 주문자 휴대전화
- 주문자 일반전화
- 수령자명
- 수령자 연락처
- 배송지 주소
- 결제 상태 상세
- 배송 상태 상세
- 주문 요청사항 또는 메모

즉, Cafe24 Admin API 응답 안에 해당 값이 있을 가능성은 있지만, 현재 코드에서는 화면에 표시할 형태로 추출하지 않습니다. 구현 단계에서는 실제 Cafe24 Admin API 응답 구조를 확인한 뒤 필드명을 확정해야 합니다.

## 5. 화면별 표시 범위 제안

### 오늘 처리 탭

오늘 처리 탭은 운영자가 매일 빠르게 훑어보는 요약 화면입니다. 개인정보를 많이 노출하지 않는 방향을 권장합니다.

추천 표시:

- 주문자명 또는 업체명
- 연락처 일부 또는 마스킹된 연락처
- 상품명
- 주문번호
- 파일명
- `file_id`
- 파일 상태
- 업로드일
- 다운로드
- 상세 보기

오늘 처리 탭에서 표시하지 않는 것을 권장:

- 전체 배송지 주소
- 전체 전화번호
- 전체 이메일
- 주문 요청사항 전체
- Cafe24 Admin API 원본 응답
- Webhook raw payload

오늘 처리 탭은 “오늘 무엇을 봐야 하는지”를 빠르게 판단하는 화면이어야 합니다. 자세한 고객 정보는 상세 화면에서 확인하도록 분리합니다.

### 파일 상세 화면

파일 상세 화면은 관리자 확인용 화면이므로 오늘 처리 탭보다 더 자세한 정보를 표시할 수 있습니다. 다만 여기에서도 원문 노출은 필요한 항목으로 제한합니다.

추천 섹션명:

`Cafe24 주문/고객 정보`

추천 표시 후보:

- 주문번호
- 주문자명
- 주문자 연락처
- 주문자 이메일
- 수령자명
- 수령자 연락처
- 배송지
- 상품명
- 옵션
- 수량
- Cafe24 주문상태
- 배송상태

주의:

- 실제 표시 필드는 Cafe24 Admin API 응답 확인 후 확정합니다.
- 전화번호와 이메일은 마스킹 정책을 먼저 정합니다.
- 배송지는 파일 검수 담당자에게 불필요할 수 있으므로 기본 접힘 또는 상세 화면 전용 표시를 검토합니다.
- CSV와 로그에는 개인정보를 기본 포함하지 않는 것을 권장합니다.

### 재업로드 탭

재업로드 탭은 고객에게 재업로드를 요청하고 새 파일을 확인하는 화면입니다.

추천 표시:

- 주문번호
- 파일명
- `file_id`
- 현재 파일 상태
- 재업로드 요청 사유
- 새 파일 ID
- 간단한 주문자명 또는 업체명

비추천 표시:

- 전체 주소
- 전체 연락처
- 전체 이메일
- 결제 정보
- 민감한 주문 요청사항 전체

### 로그 확인 탭

로그 확인 탭은 운영 기록과 문제 원인을 보는 곳입니다. 개인정보는 최소화해야 합니다.

권장 기준:

- Webhook 로그에는 raw payload 전체를 표시하지 않습니다.
- 다운로드 로그와 CSV에는 개인정보를 기본 포함하지 않습니다.
- 교정확인 이력에는 파일 처리에 필요한 문구와 상태 중심으로 표시합니다.
- 고객 연락처, 주소, 이메일은 로그 화면 기본 목록에 포함하지 않습니다.

## 6. 디자이너/파일 검수 담당 기준 표시 정보

디자이너 또는 파일 검수 담당자에게 필요한 정보 후보는 다음과 같습니다.

- 파일명
- `file_id`
- 주문번호
- 상품명
- 상품 옵션
- 수량
- 파일 상태
- 재업로드 필요 여부
- 간단한 고객명 또는 업체명

디자이너에게 불필요하거나 최소화할 정보 후보는 다음과 같습니다.

- 전체 배송지 주소
- 전체 전화번호
- 전체 이메일
- 결제 정보
- 배송 상세 정보
- 민감한 고객 요청사항
- Cafe24 회원 ID
- Webhook payload

정리 기준:

디자이너 전용 화면을 별도로 만들기 전까지는 `/admin` 파일 상세 화면에서 개인정보 노출 범위를 신중하게 조정합니다. 파일 검수에 직접 필요한 정보는 상품명, 옵션, 수량, 파일 상태 중심으로 보고, 연락처와 주소는 관리자 확인용으로 분리하는 것이 좋습니다.

## 7. 데이터 확보 방식 비교

### A안. 조회 시 Cafe24 Admin API로 주문정보를 가져와 표시

장점:

- 최신 주문자, 배송지, 주문 상태를 볼 수 있습니다.
- Supabase DB에 개인정보를 별도로 저장하지 않아도 됩니다.
- 개인정보 보관 범위를 늘리지 않는 방향입니다.

단점:

- Cafe24 API 호출 실패 시 표시할 수 없습니다.
- 응답 속도가 느릴 수 있습니다.
- OAuth token 만료나 Cafe24 API 오류의 영향을 받습니다.
- 실제 응답 필드명이 주문 유형에 따라 다를 수 있어 파싱 보완이 필요합니다.

### B안. Webhook 또는 주문 조회 후 필요한 고객정보 일부를 Supabase에 저장

장점:

- `/admin` 화면에서 빠르게 조회할 수 있습니다.
- Cafe24 API 일시 장애와 관계없이 기본 정보를 볼 수 있습니다.
- 오늘 처리 탭에서 요약 표시하기 쉽습니다.

단점:

- 개인정보 저장 범위가 늘어납니다.
- 보관 기간, 마스킹, 접근 권한, 삭제 정책이 필요합니다.
- CSV, 로그, 백업에 개인정보가 섞이지 않도록 추가 관리가 필요합니다.

### C안. `files`와 분리된 별도 `order_customer_info` 테이블에 최소 정보 저장

장점:

- 파일 메타데이터와 고객정보를 분리할 수 있습니다.
- 권한, 마스킹, 보관 정책을 별도로 설계하기 좋습니다.
- 나중에 역할별 화면을 만들 때 확장하기 쉽습니다.

단점:

- DB 스키마와 동기화 로직이 추가됩니다.
- 개인정보 저장 정책을 먼저 정해야 합니다.
- 주문정보 갱신 시점과 최신성 기준을 별도로 관리해야 합니다.

### 현재 프로젝트 1차 추천

1차 구현은 A안, 즉 “파일 상세 화면에서 필요할 때 Cafe24 Admin API로 주문/고객 정보를 조회해 표시”하는 방식을 우선 검토하는 것이 좋습니다.

이유:

- DB 스키마 변경 없이 시작할 수 있습니다.
- 개인정보를 Supabase에 새로 저장하지 않아도 됩니다.
- 현재 이미 Cafe24 주문 조회 테스트 기능과 주문 품목 조회 로직이 있습니다.
- 개인정보 표시 범위를 실제 운영 화면에서 확인한 뒤, 정말 자주 필요한 최소 정보만 나중에 저장할 수 있습니다.

다만 구현 전에는 실제 Cafe24 Admin API 응답에서 주문자명, 연락처, 이메일, 수령자, 배송지, 상품 옵션 필드명이 어떻게 내려오는지 확인해야 합니다.

## 8. 1차 구현 추천 범위

1차 구현은 너무 넓게 잡지 않는 것을 권장합니다.

1차 구현 후보:

- `/admin` 파일 상세 화면에 `Cafe24 주문/고객 정보` 섹션 추가
- `files.order_id`가 연결된 파일에만 표시
- Cafe24 Admin API로 주문정보를 조회해 가능한 정보만 표시
- 주문자명, 연락처, 이메일, 상품명, 옵션, 배송지 정보를 후보로 표시
- 오늘 처리 탭에는 고객명과 상품명 정도만 추후 검토
- CSV에는 개인정보 미포함
- Webhook 로직 변경 없음
- DB 스키마 변경 없음

1차에서 제외:

- 역할별 권한 분리
- 디자이너 전용 화면
- 개인정보 DB 저장
- 다운로드 로그에 고객정보 포함

- 주소 기반 필터

- Cafe24 운영 스킨 변경

## 9. 개인정보 및 보안 기준

개인정보 표시 기준은 다음을 권장합니다.

- 고객정보는 필요한 화면에만 최소 노출합니다.
- 오늘 처리 탭에는 전체 주소를 표시하지 않습니다.
- 로그와 CSV에는 개인정보를 기본 포함하지 않습니다.
- Webhook raw payload 전체를 화면에 표시하지 않습니다.
- Cafe24 Admin API 응답 전체를 화면에 표시하지 않습니다.
- `access_token`, `refresh_token`, authorization, secret, signature는 절대 표시하지 않습니다.
- 전화번호와 이메일은 필요한 경우 마스킹 표시를 검토합니다.
- 주소는 파일 상세 화면에서만 표시하는 방향을 우선 검토합니다.
- 디자이너가 볼 필요 없는 개인정보는 최소화합니다.
- 개인정보 저장이 필요해지면 별도 정책 수립 후 진행합니다.

마스킹 예시 정책 후보:

| 항목 | 표시 후보 |
|---|---|
| 휴대전화 | 가운데 자리 마스킹 또는 뒷자리만 표시 |
| 이메일 | 아이디 일부와 도메인 일부 마스킹 |
| 주소 | 오늘 처리 탭에는 미표시, 상세 화면에서만 표시 |
| 주문자명 | 전체 표시 또는 일부 마스킹 정책 선택 |
| 회원 ID | 필요 시 일부 마스킹 |

섹션명:

`Cafe24 주문/고객 정보`

표시 후보:

- 주문번호
- 주문자명
- 주문자 연락처
- 주문자 이메일
- 수령자명
- 수령자 연락처
- 배송지
- 상품명
- 옵션
- 수량
- Cafe24 주문상태
- 배송상태

화면 구성 방향:

- 파일 상세 정보 아래 또는 주문번호 연결 영역 주변에 배치합니다.
- 개인정보가 포함되는 섹션임을 알 수 있게 짧은 안내 문구를 넣습니다.
- Cafe24 주문 조회 실패 시 관리자에게 이해 가능한 오류 문구를 보여줍니다.
- 주소나 연락처는 너무 넓게 펼쳐지지 않게 카드 또는 `dl` 형태로 정리합니다.
- 긴 상품 옵션은 줄바꿈되도록 처리합니다.
- 응답 원본 JSON은 표시하지 않습니다.

## 11. 오류 및 빈 상태 처리 문구

구현 단계에서 아래 상황별 문구를 사용할 수 있습니다.

| 상황 | 문구 후보 |
|---|---|
| 주문번호가 연결되지 않은 파일 | 주문번호가 연결되지 않아 고객정보를 확인할 수 없습니다. |
| Cafe24 주문 조회 실패 | Cafe24 주문 정보를 불러오지 못했습니다. 설정 탭의 OAuth 상태를 확인해 주세요. |
| 주문은 조회됐지만 고객정보 없음 | 고객정보 일부가 Cafe24 응답에 포함되지 않았습니다. |
| OAuth token 만료 또는 Cafe24 API 오류 | Cafe24 API 연결 상태를 확인한 뒤 다시 시도해 주세요. |
| 네트워크 오류 | 일시적으로 주문 정보를 불러오지 못했습니다. 잠시 후 다시 시도해 주세요. |

오류 메시지에는 access token, refresh token, authorization, API 응답 원문을 포함하지 않습니다.
