# Cafe24 주문조회 화면 파일 상태 조회 연결 설계 문서

- 작성 기준: 2026-07-06
- 대상 기능: Cafe24 주문조회/주문상세 화면에서 파일 상태 조회 페이지 연결
- 문서 성격: 구현 전 설계 문서

## 1. 문서 목적

이 문서는 고객이 Cafe24 주문완료 화면, 주문조회 화면, 주문상세 화면, 문자, 카카오톡, 이메일 안내를 통해 `/file-status`로 이동해 본인이 업로드한 인쇄용 파일의 처리 상태를 확인할 수 있도록 연결 방식을 정리하기 위한 문서입니다.

이번 문서는 설계 문서입니다. 앱 코드, API, DB 로직, Webhook 로직, Cafe24 스마트디자인 코드는 수정하지 않습니다.

## 2. 현재 상태 요약

- 고객용 `/file-status` 페이지는 이미 구현되어 있습니다.
- 고객은 주문번호와 `file_id`를 함께 입력해 파일 상태를 조회할 수 있습니다.
- `file_id` 단독 조회는 허용하지 않습니다.
- Cafe24 품목별 주문번호 끝에 `-01`, `-02`가 붙어 있어도 상위 주문번호 기준으로 조회할 수 있습니다.
- 고객 화면에는 다운로드 버튼을 제공하지 않습니다.
- 고객 화면과 API 응답에는 `storage_path`, signed URL, `token_hash`, raw token, Webhook raw payload를 노출하지 않습니다.
- 현재 남은 과제는 고객이 `/file-status` 페이지를 쉽게 찾도록 Cafe24 화면 또는 고객 안내문에서 자연스럽게 연결하는 것입니다.

## 4. 1차 추천 방향

1차 구현은 Cafe24 화면을 크게 변경하지 않고, 아래 방식부터 적용하는 것을 추천합니다.

- Cafe24 주문상세 화면 또는 주문조회 안내 영역에 “업로드 파일 상태 조회하기” 링크를 제공합니다.
- 링크는 `/file-status`로 이동합니다.
- 고객은 주문번호와 업로드 파일 ID를 직접 입력합니다.
- URL에 `order_id`와 `file_id`를 자동으로 붙이는 방식은 1차에서 제외합니다.
- 문자, 카카오톡, 이메일 안내문에도 `/file-status` 링크를 함께 넣습니다.

추천 이유는 다음과 같습니다.

- 현재 `/file-status` 구조와 바로 맞습니다.
- 고객 화면에 민감정보를 추가로 노출하지 않습니다.
- Cafe24 템플릿 변수나 품목 옵션 자동 파싱에 덜 의존합니다.
- URL 공유, 브라우저 기록, 외부 전달로 인한 주문정보 노출 가능성을 줄일 수 있습니다.
- 자동 입력 방식은 Cafe24 템플릿 구조를 확인한 뒤 2차 개선으로 검토하는 편이 안전합니다.

## 7. URL 파라미터 자동 입력 방식 검토

예상 URL 형식은 다음과 같습니다.

```text
/file-status?order_id=20260704-0000014&file_id=...
```

### 장점

- 고객이 주문번호와 `file_id`를 직접 입력하지 않아도 됩니다.
- 주문상세 화면에서 바로 상태 조회 결과에 가까운 경험을 줄 수 있습니다.

### 위험과 주의사항

- URL에 주문번호와 `file_id`가 노출됩니다.
- 고객이 링크를 공유하거나 브라우저 기록에 남길 수 있습니다.
- Cafe24 템플릿에서 주문번호와 품목 옵션의 `file_id`를 안정적으로 가져올 수 있는지 확인해야 합니다.
- 주문상세에 여러 품목이 있는 경우 어떤 `file_id`를 연결할지 품목별 처리가 필요합니다.

### 결론

1차에서는 URL 파라미터 자동 입력을 제외합니다. 고객이 직접 주문번호와 `file_id`를 입력하는 방식이 더 안전합니다.

URL 파라미터 자동 입력은 2차 개선 후보로 문서화하고, Cafe24 템플릿 변수와 품목 옵션 구조를 확인한 뒤 다시 검토합니다.

## 8. Cafe24 스마트디자인 삽입 방식 검토

이번 문서에서는 실제 Cafe24 스마트디자인 코드를 작성하거나 수정하지 않습니다. 구현 단계 전에 아래 항목을 확인해야 합니다.

### 확인 필요 항목

- 주문완료 화면 템플릿 경로
- 주문조회 목록 템플릿 경로
- 주문상세 화면 템플릿 경로
- 상품 옵션 영역에 업로드 파일 ID가 표시되는지 여부
- Cafe24 템플릿 변수로 상위 주문번호를 가져올 수 있는지 여부
- Cafe24 템플릿 변수로 품목별 주문번호를 가져올 수 있는지 여부
- Cafe24 템플릿 변수로 품목 옵션 값을 가져올 수 있는지 여부
- 단순 HTML 안내 박스만 넣을지 여부
- JS로 `file_id`를 찾아 안내 영역을 자동 생성할지 여부

### 1차 권장 삽입 방식

1차에서는 JS 자동 파싱보다 단순 안내 박스와 `/file-status` 링크 제공을 우선 추천합니다.

이유는 다음과 같습니다.

- Cafe24 주문상세 화면의 기존 레이아웃을 덜 건드립니다.
- 품목 옵션 구조가 바뀌어도 안내 링크 자체는 유지됩니다.
- 고객이 직접 주문번호와 `file_id`를 확인해 입력하므로 자동 매칭 오류 위험이 낮습니다.

## 9. 1차 구현 범위 제안

### 1차 구현

- Cafe24 주문상세 화면 또는 주문완료/주문조회 안내 영역에 파일 상태 조회 안내 박스 추가
- `/file-status` 링크 버튼 제공
- 고객은 주문번호와 `file_id`를 직접 입력
- 문자/카카오톡/이메일 안내문에 `/file-status` 링크 추가
- 고객 다운로드 버튼 제공 없음
- URL 파라미터 자동 입력 없음

### 1차에서 제외

- URL에 `order_id`와 `file_id`를 넣어 자동 입력
- 고객 다운로드 기능
- Cafe24 주문 옵션 자동 파싱 JS
- 고객 인증 시스템
- SMS 자동 발송
- 이메일 자동 발송
- 재업로드 링크 자동 표시
- Cafe24 앱 설정 화면 자동 삽입 기능

## 10. 2차 개선 후보

- `/file-status?order_id=...&file_id=...` 자동 입력 지원
- Cafe24 주문상세에서 `file_id`를 자동으로 읽어 상태 조회 버튼 생성
- 주문상세에서 파일별 “상태 조회” 버튼을 품목별로 표시
- 재업로드 필요 상태일 때 고객 안내 문구 강화
- 주문완료 안내 영역에 상태 조회 링크 자동 삽입
- 조회 시도 제한 또는 rate limit 추가
- 고객용 파일 상태 조회 FAQ 추가

## 11. 보안 기준

- 고객 화면에는 다운로드 링크를 제공하지 않습니다.
- `storage_path`, signed URL, `token_hash`, raw token은 절대 노출하지 않습니다.
- URL 파라미터 자동 입력은 1차에서 제외합니다.
- 조회 실패 시 어떤 값이 맞고 어떤 값이 틀렸는지 구분해서 알려주지 않습니다.
- `file_id` 단독 조회를 허용하지 않습니다.
- 주문번호와 `file_id` 조합 조회 기준을 유지합니다.
- 재업로드 링크는 기존 `/reupload?token=...` 방식을 유지합니다.
- 재업로드 raw token 원문을 DB에 저장하지 않는 원칙을 유지합니다.
- `token_hash`만 저장하는 현재 방식을 유지합니다.
- Cafe24 화면에는 민감정보를 직접 출력하지 않습니다.
- Admin API 응답 전체 또는 Webhook raw payload를 고객 화면에 노출하지 않습니다.

## 12. 운영 정책 결정 필요 항목

- Cafe24 스마트디자인 수정은 운영자가 직접 할지, Codex가 HTML 스니펫 초안을 만들지

- 상위 주문번호와 품목별 주문번호를 모두 입력 가능하다고 안내할지
- 문자/카카오톡/이메일 안내문을 실제 운영에 사용할지
- `/file-status` 링크를 Cafe24 푸터 또는 고객센터에도 넣을지
- URL 파라미터 자동 입력을 2차에서 허용할지
- 조회 시도 제한 또는 rate limit을 언제 도입할지
