# 페르패키지 파일 업로드 앱 직원용 실무 체크리스트

- 작성 기준: 2026-07-06
- 대상 화면: `/admin` 관리자 화면
- 문서 성격: 직원용 짧은 실무 체크리스트
- 자세한 설명: `docs/ADMIN_OPERATION_MANUAL.md`

## 목적

- 매일 업로드 파일을 빠르게 확인하기 위한 체크리스트입니다.
- 화면에서 무엇을 확인하고, 어떤 버튼을 눌러야 하는지 중심으로 정리합니다.
- 개인정보와 민감정보는 원문으로 복사하거나 공유하지 않습니다.

## 1. 매일 시작할 때

- [ ] `/admin`에 로그인한다.
- [ ] 오늘 처리 탭을 연다.
- [ ] 최근 업로드 파일을 확인한다.
- [ ] 확인 전 파일을 우선 확인한다.
- [ ] 재업로드 완료 파일이 있는지 확인한다.
- [ ] 확인 필요 Webhook이 있는지 확인한다.

## 2. 최근 업로드 파일 처리 순서

오늘 처리 탭은 빠르게 훑어보는 요약 화면입니다.

- [ ] 파일명을 확인한다.
- [ ] `file_id`를 확인한다.
- [ ] Cafe24 주문번호가 연결되어 있는지 확인한다.
- [ ] 필요한 경우 다운로드 버튼으로 파일을 다운로드한다.
- [ ] 상세 보기를 눌러 `/admin?tab=files&file_id=<file_id>`로 이동한다.
- [ ] 파일 상세 화면에서 Cafe24 주문 정보를 확인한다.
- [ ] 파일을 검수한 뒤 파일 상태를 변경한다.

주의: 오늘 처리 탭에서는 상태를 변경하지 않습니다. 상태 변경은 상세 보기로 들어간 파일 찾기 탭의 파일 상세 화면에서 처리합니다.

## 3. 파일 상세에서 꼭 확인할 것

- [ ] 파일명과 `file_id`가 맞는지 확인한다.
- [ ] Cafe24 주문번호가 맞는지 확인한다.
- [ ] 상품명과 상품 옵션을 확인한다.
- [ ] 업로드 파일 ID가 현재 `file_id`와 맞는지 확인한다.
- [ ] Cafe24 주문 정보에서 품목 수량을 확인한다.
- [ ] Cafe24 주문 정보에서 품목 상태를 확인한다.
- [ ] 필요한 경우 파일을 다운로드해 실제 파일을 연다.

주의:

- 주문자명, 전화번호, 이메일, 주소는 현재 파일 상세 화면에서 보지 않습니다.
- 고객명이나 회원 ID처럼 보이는 내부 정보는 직원 업무 화면에 표시하지 않는 기준입니다.
- 고객 개인정보처럼 보이는 값은 캡처하거나 외부로 공유하지 않습니다.

## 4. 파일 검수 후 상태 변경 기준

| 상태값 | 화면 의미 | 언제 선택하는지 |
|---|---|---|
| `uploaded_pending` | 업로드됨 / 확인 전 | 고객이 업로드했고 아직 파일을 열어보지 않은 상태 |
| `reviewing` | 파일 확인 중 | 파일을 검수 중이거나 내부 확인이 필요한 상태 |
| `approved` | 파일 확인 완료 | 파일 검수가 끝났고 제작 진행이 가능한 상태 |
| `need_reupload` | 재업로드 요청 | 파일에 문제가 있어 고객 수정본이 필요한 상태 |
| `replaced` | 새 파일로 교체됨 | 새 파일이 들어와 기존 파일을 이전 파일로 남길 때 |
| `archived` | 보관 처리 | 더 이상 처리하지 않고 기록으로 보관할 때 |

주의:

- `approved`는 파일을 실제로 확인한 뒤에만 선택한다.
- 재업로드 새 파일도 직접 다운로드해 확인한 뒤 상태를 바꾼다.
- 기존 파일은 자동으로 `replaced` 처리되지 않는다.

## 5. 주문번호가 없을 때

- [ ] Cafe24 주문상세에 업로드 파일 ID가 들어갔는지 확인한다.
- [ ] `/admin` 파일 찾기 탭에서 Cafe24 주문 조회 테스트를 실행한다.
- [ ] 주문번호로 파일 찾기를 실행한다.
- [ ] `file_id`로 파일 찾기를 실행한다.
- [ ] Cafe24 주문상세의 업로드 파일 ID와 `/admin`의 `file_id`가 같은지 다시 확인한다.
- [ ] 필요하면 주문번호를 수동 연결한다.

주의: 다른 주문번호가 이미 연결된 파일은 임의로 덮어쓰지 않습니다.

## 6. 재업로드가 필요할 때

- [ ] 파일 상세 화면에서 파일 상태를 `need_reupload`로 변경한다.
- [ ] 재업로드 탭으로 이동한다.
- [ ] 해당 `file_id` 기준으로 재업로드 요청을 생성한다.
- [ ] 재업로드 안내문을 복사한다.
- [ ] 필요하면 재업로드 링크를 생성한다.
- [ ] 고객에게 안내문과 링크를 전달한다.
- [ ] 고객이 새 파일을 올리면 오늘 처리 탭 또는 재업로드 탭에서 재업로드 완료 파일을 확인한다.
- [ ] 새 파일 상세 보기로 들어간다.
- [ ] 새 파일을 다운로드하고 검수한다.
- [ ] 새 파일 상태를 `reviewing` 또는 `approved`로 직접 변경한다.
- [ ] 기존 파일을 교체 처리해야 하면 기존 파일 상태를 `replaced`로 직접 변경한다.

재업로드 주의사항:

- [ ] 재업로드 완료 후 기존 파일은 자동 삭제되지 않는다.
- [ ] 기존 파일은 자동으로 `replaced` 처리되지 않는다.
- [ ] 새 파일은 자동으로 `approved` 처리되지 않는다.
- [ ] 반드시 관리자가 직접 확인 후 상태를 바꾼다.
- [ ] 재업로드 링크의 token 원문을 문서나 메신저에 불필요하게 남기지 않는다.

## 7. 다운로드 후 확인

- [ ] 파일이 정상 다운로드되는지 확인한다.
- [ ] 다운로드 파일명이 이상하지 않은지 확인한다.
- [ ] 다운로드 후 필요하면 로그 확인 탭에서 다운로드 로그를 확인한다.
- [ ] signed URL 원문은 복사하거나 공유하지 않는다.

## 8. 고객이 파일 상태를 물어볼 때

- [ ] 고객에게 `/file-status`에서 주문번호와 업로드 파일 ID를 입력하라고 안내한다.
- [ ] 품목별 주문번호 끝에 `-01`, `-02`가 붙어 있어도 조회 가능하다고 안내한다.
- [ ] 상태 조회 화면에서는 파일 다운로드를 제공하지 않는다고 안내한다.
- [ ] 재업로드가 필요한 경우 안내받은 재업로드 링크를 사용하라고 안내한다.

주의: 직원이 고객 대신 민감정보를 입력하거나 token, signed URL, 내부 경로를 공유하지 않습니다.

## 9. Webhook 확인 필요 상태

정상 상태:

- `auto_linked`: 자동 연결 완료
- `already_linked`: 이미 연결됨

확인 필요 상태:

- `received`
- `no_order_id`
- `no_file_id`
- `file_not_found`
- `conflict_order_id`
- `failed`

확인 필요 상태가 있으면 관리자 또는 개발 담당자에게 공유합니다.

특히 `conflict_order_id`는 다른 주문번호가 이미 연결된 상황일 수 있습니다. 절대 임의로 덮어쓰지 않습니다.

## 10. 절대 하지 말아야 할 것

- [ ] `file_id`와 주문번호를 확인하지 않고 수동 연결하지 않는다.
- [ ] 다른 주문번호가 이미 연결된 파일을 임의로 덮어쓰지 않는다.
- [ ] 파일을 검수하지 않고 `approved`로 바꾸지 않는다.
- [ ] 재업로드 새 파일을 확인하지 않고 `approved`로 바꾸지 않는다.
- [ ] 기존 파일을 임의로 삭제하지 않는다.
- [ ] Webhook 오류를 임의로 수정하려고 하지 않는다.
- [ ] 고객 개인정보를 캡처해 외부로 공유하지 않는다.
- [ ] signed URL, token, secret, key, raw payload를 복사해서 공유하지 않는다.

## 11. 원문 공유 금지 항목

아래 항목은 원문으로 복사하거나 공유하지 않습니다.

- `storage_path`
- signed URL 원문
- raw token
- `token_hash`
- `access_token`
- `refresh_token`
- `authorization`
- `bearer`
- `client_secret`
- `secret`
- `signature`
- `password`
- `cookie`
- `x-api-key`
- `api-key`
- `api_key`
- Supabase service role key
- Naver Object Storage access key
- Naver Object Storage secret key
- Cafe24 OAuth token
- Webhook raw payload 전체
- Cafe24 Admin API raw 응답 전체
- 고객 이름 원문
- 고객 전화번호 원문
- 고객 이메일 원문
- 고객 주소 원문
- 배송 메시지 원문
- 주문 요청사항 원문

- [ ] 파일명
- [ ] 현재 파일 상태
- [ ] 어떤 탭에서 문제가 생겼는지

- [ ] Webhook `processed_status`, 확인 가능한 경우

보고할 때도 민감정보 원문은 포함하지 않습니다.

## 13. 하루 마감 전 확인

- [ ] 오늘 업로드된 파일 중 확인 전 상태가 남아 있는지 확인한다.
- [ ] 재업로드 완료 파일 중 미확인 파일이 남아 있는지 확인한다.
- [ ] 다운로드 실패 로그가 있는지 확인한다.
- [ ] `failed` 또는 `conflict_order_id` Webhook 로그가 있는지 확인한다.
- [ ] 처리한 파일의 상태가 올바르게 변경됐는지 확인한다.
