# 고객용 파일 상태 조회 페이지 설계 문서

- 작성 기준: 2026-07-05
- 대상 기능: 고객용 파일 상태 조회 페이지
- 문서 성격: 구현 전 설계 문서

## 1. 기능 목적

고객용 파일 상태 조회 페이지는 고객이 본인이 업로드한 파일의 현재 처리 상태를 직접 확인할 수 있게 하는 기능입니다.

목표는 다음과 같습니다.

- 고객이 파일 접수 여부를 직접 확인할 수 있게 합니다.
- 고객센터로 들어오는 "파일이 접수됐나요?", "확인 중인가요?", "재업로드가 필요한가요?" 같은 문의를 줄입니다.
- 파일이 접수됐는지, 확인 중인지, 재업로드가 필요한지, 확인 완료인지 부드러운 문구로 안내합니다.
- 관리자 화면의 상세 정보나 내부 이력, 민감정보는 고객에게 노출하지 않습니다.

예상 페이지 URL 후보는 다음과 같습니다.

| 후보 | 설명 | 비고 |
|---|---|---|
| `/file-status` | 기능 목적이 명확함 | 1차 구현 추천 |
| `/status` | 짧고 입력하기 쉬움 | 다른 상태 페이지와 충돌 가능성 검토 필요 |

이번 문서는 설계 문서입니다. 실제 페이지, API, DB 변경은 구현하지 않습니다.

## 2. 고객 조회 방식 후보

### A안: file_id 단독 조회

고객이 업로드 후 받은 `file_id`만 입력해 상태를 조회하는 방식입니다.

| 구분 | 내용 |
|---|---|
| 장점 | 입력값이 1개라 사용이 쉽고 구현이 단순합니다. |
| 단점 | `file_id`만 알면 조회할 수 있으므로 보안 안내와 조회 제한이 필요합니다. |
| 추천 여부 | 1차 구현에서는 비추천입니다. 내부 테스트나 관리자 보조 기능으로는 검토할 수 있습니다. |

### B안: 주문번호 + file_id 조회

고객이 Cafe24 주문번호와 `file_id`를 함께 입력해 상태를 조회하는 방식입니다.

| 구분 | 내용 |
|---|---|
| 장점 | `file_id` 단독 조회보다 안전합니다. Cafe24 주문번호와 업로드 파일 ID를 함께 확인할 수 있습니다. |
| 단점 | 고객이 입력해야 할 값이 2개라 조금 번거롭습니다. |
| 추천 여부 | 1차 구현 추천안입니다. 현재 관리자 주문번호 연결 흐름과도 가장 잘 맞습니다. |

### C안: 조회 전용 토큰 링크

고객에게 별도 조회 링크를 제공하고, 링크 접속만으로 상태를 확인하는 방식입니다.

| 구분 | 내용 |
|---|---|
| 장점 | 고객 사용성이 가장 좋습니다. 입력 없이 바로 상태 확인이 가능합니다. |
| 단점 | 토큰 생성, 만료, 저장, 재발급 정책이 추가로 필요합니다. |
| 추천 여부 | 2차 이후 기능으로 검토합니다. 재업로드 링크 구조를 참고할 수 있습니다. |

## 3. 1차 구현 추천 방향

현재 프로젝트 기준 1차 구현은 `주문번호 + file_id 조합 조회`를 추천합니다.

추천 이유는 다음과 같습니다.

- `file_id`만 아는 제3자가 조회하는 위험을 줄일 수 있습니다.
- Cafe24 주문번호와 업로드 파일 ID를 함께 확인할 수 있습니다.
- 고객이 주문 완료 후 안내받기 쉬운 값 조합입니다.
- 관리자 화면의 `files.order_id` 연결 흐름과 맞습니다.
- 현재 DB 구조를 크게 바꾸지 않고 설계할 수 있습니다.

단, 실제 구현 전에는 현재 `files` 테이블 구조와 주문번호 저장 방식이 이 조회 방식에 충분히 맞는지 확인해야 합니다.

## 5. 파일 상태값별 고객 표시 문구

아래 문구는 고객이 불안하지 않게 부드러운 톤으로 작성합니다.

주의: "제작 확정", "인쇄 확정", "납기 확정"처럼 과도하게 확정적인 표현은 사용하지 않습니다.

| 파일 상태값 | 고객 표시 문구 |
|---|---|
| `uploaded_pending` | 파일이 정상 접수되었습니다. 담당자가 순차적으로 확인할 예정입니다. |
| `reviewing` | 파일을 확인 중입니다. 인쇄 가능 여부를 검토하고 있습니다. |
| `approved` | 파일 확인이 완료되었습니다. 제작 진행이 가능한 상태입니다. |
| `need_reupload` | 파일 수정 또는 재업로드가 필요합니다. 안내받은 링크로 수정 파일을 다시 업로드해 주세요. |
| `replaced` | 새 파일로 교체된 이전 파일입니다. 최신 파일 기준으로 확인해 주세요. |
| `archived` | 처리가 완료되었거나 보관 처리된 파일입니다. |

## 7. 고객 화면 구성안

### 상단

- 페이지 제목: 파일 상태 조회
- 안내문: 주문번호와 업로드 파일 ID를 입력하면 파일 접수 및 확인 상태를 조회할 수 있다는 설명

### 조회 입력 영역

- 주문번호 입력칸
- `file_id` 입력칸
- 조회 버튼

### 조회 결과 영역

- 파일명
- 파일 상태
- 업로드일
- 주문번호
- 고객 안내 문구
- 재업로드 필요 여부
- 재업로드 버튼, 가능한 경우에만 표시

### 오류/빈 상태

| 상황 | 고객 표시 방향 |
|---|---|
| 입력값이 비어 있음 | 주문번호와 업로드 파일 ID를 모두 입력해 달라고 안내 |
| 조회 결과가 없음 | 입력한 정보로 파일을 찾을 수 없다고 안내 |
| 주문번호와 file_id가 일치하지 않음 | 입력한 정보로 파일을 찾을 수 없다고 안내 |
| 서버 오류 | 잠시 후 다시 시도하거나 고객센터에 문의해 달라고 안내 |
| 재업로드 링크가 만료됨 | 재업로드 링크가 만료되어 다시 안내가 필요하다고 안내 |

보안상 조회 실패 시 어떤 값이 맞고 어떤 값이 틀렸는지 구체적으로 알려주지 않습니다.

## 8. API 설계 후보

이번 작업에서는 API를 구현하지 않습니다. 구현 단계에서 검토할 후보만 정리합니다.

### 추천 API

- `POST /api/file-status`

`GET`보다 `POST`를 추천합니다. 주문번호와 `file_id`가 URL query에 남는 것을 줄일 수 있기 때문입니다.

### 입력값 후보

| 필드 | 설명 |
|---|---|
| `order_id` | Cafe24 주문번호 |
| `file_id` | 업로드 파일 ID |

### 응답값 후보

| 필드 | 설명 |
|---|---|
| `success` | 조회 성공 여부 |
| `status` | 내부 파일 상태값 |
| `status_label` | 고객용 상태명 |
| `customer_message` | 고객에게 보여줄 안내 문구 |
| `file_name` | 파일명 |
| `uploaded_at` | 업로드일 |
| `order_id` | 주문번호, 노출 정책 결정 필요 |
| `reupload_required` | 재업로드 필요 여부 |
| `reupload_available` | 재업로드 버튼 표시 가능 여부 |
| `reupload_url` | 직접 노출 여부 별도 검토 |
| `error_code` | 오류 코드 |
| `error_message` | 고객용 오류 문구 |

### API 보안 주의

응답에 아래 값이 포함되면 안 됩니다.

- `storage_path`
- signed URL
- `token_hash`
- raw token
- 내부 메모
- Webhook payload 전체
- 관리자용 이력 전체

## 9. DB 조회 기준 설계

현재 구조를 전제로 아래 기준을 검토합니다.

1. `files` 테이블에서 `file_id`로 파일을 조회합니다.
2. 입력한 `order_id`가 파일의 `order_id`와 일치하는지 확인합니다.
3. `customer_type`이 `cafe24-reupload`인 경우 원본 파일과의 관계를 표시할지 검토합니다.
4. `file_reupload_requests`에서 해당 `original_file_id` 또는 `new_file_id` 관련 요청 상태를 확인합니다.
5. 재업로드 요청이 `requested` 상태이면 고객에게 재업로드 가능 안내를 표시합니다.
6. 재업로드 요청이 `uploaded` 상태이면 재업로드 완료 안내를 표시합니다.
7. 재업로드 요청이 `completed` 상태이면 처리 완료 안내를 표시합니다.

실제 컬럼명과 관계는 코드 및 운영 DB 스키마를 확인한 뒤 구현 단계에서 확정해야 합니다.

## 10. 보안 기준

- 조회 방식은 `file_id` 단독보다 주문번호 + `file_id` 조합을 우선 검토합니다.
- 조회 실패 시 어떤 값이 맞고 어떤 값이 틀렸는지 구체적으로 알려주지 않습니다.
- 존재하지 않는 `file_id`인지, 주문번호가 틀린 것인지 구분해서 알려주지 않습니다.
- 너무 많은 조회 시도를 제한하는 rate limit 검토가 필요합니다.
- 고객 화면에서는 다운로드 링크를 기본 제공하지 않습니다.
- 파일 다운로드는 관리자 화면에서만 처리하는 것을 기본으로 합니다.
- 재업로드 링크는 기존 `/reupload?token=...` 방식을 재사용하되, 토큰 원문 저장 금지 원칙을 유지합니다.
- `token_hash`만 저장하는 현재 방식을 유지합니다.
- raw token, `token_hash`, signed URL 원문은 노출하지 않습니다.

## 11. 구현 전 운영 정책 결정 필요 항목

아래 항목은 구현 전에 운영 정책으로 먼저 결정해야 합니다.

- 고객이 주문번호 + `file_id`로 조회하게 할지
- `file_id` 단독 조회도 허용할지
- 고객 화면에서 파일명을 전체 노출할지 일부 마스킹할지
- 주문번호를 전체 노출할지 일부 마스킹할지
- 고객에게 다운로드 버튼을 제공할지 여부
- 재업로드 링크를 상태 조회 화면에서 바로 보여줄지 여부
- 재업로드 요청이 여러 번 있을 때 어떤 요청을 최신으로 볼지
- 상태 조회 페이지를 Cafe24 상품상세, 주문완료, 문자 안내 중 어디에 연결할지
- 조회 시도 제한을 어떻게 둘지

- 관리자 이력 전체 노출

## 13. 구현 단계 테스트 기준

구현 단계에서는 아래 항목을 확인합니다.

- [ ] 올바른 주문번호 + `file_id` 조회 성공
- [ ] 틀린 주문번호 + 맞는 `file_id` 조회 실패
- [ ] 맞는 주문번호 + 틀린 `file_id` 조회 실패
- [ ] 없는 `file_id` 조회 실패
- [ ] `uploaded_pending` 상태 문구 확인
- [ ] `reviewing` 상태 문구 확인
- [ ] `approved` 상태 문구 확인
- [ ] `need_reupload` 상태 문구 확인
- [ ] `replaced` 상태 문구 확인
- [ ] `archived` 상태 문구 확인
- [ ] 재업로드 `requested` 상태 안내 확인
- [ ] 재업로드 `uploaded` 상태 안내 확인
- [ ] 만료된 재업로드 링크 안내 확인
- [ ] API 응답에 `storage_path` 미포함 확인
- [ ] API 응답에 signed URL 미포함 확인
- [ ] API 응답에 `token_hash`와 raw token 미포함 확인
- [ ] 고객 화면에 관리자용 이력 미노출 확인
- [ ] 모바일 화면 확인

## 14. 구현 단계에서 수정하면 안 되는 기본 원칙

- 관리자용 `/admin` 상세 정보는 고객 화면에 그대로 노출하지 않습니다.
- Webhook 자동 연결 로직은 상태 조회 기능과 분리합니다.
- 파일 다운로드 기능은 기본적으로 관리자 화면에만 둡니다.
- 재업로드 토큰은 원문 저장하지 않습니다.
- 고객 화면에는 내부 저장 경로, signed URL, raw payload, 내부 메모를 표시하지 않습니다.
