# 페르패키지 Cafe24 파일 업로드 앱 관리자 운영 매뉴얼

작성 기준: 2026-07-06
대상 화면: `/admin`
문서 성격: 관리자용 상세 운영 매뉴얼

## 1. 문서 목적

이 문서는 페르패키지 Cafe24 파일 업로드 앱의 `/admin` 화면에서 업로드 파일을 확인하고 처리하는 방법을 설명합니다.

관리자는 이 문서를 기준으로 고객 업로드 파일 확인, 주문번호 연결, 파일 다운로드, 파일 상태 변경, 재업로드 요청, 교정확인, Webhook 로그 확인, 다운로드 로그 확인을 진행합니다.

개발자용 설명보다 실제 화면에서 무엇을 눌러야 하는지, 어떤 값을 확인해야 하는지, 어떤 정보는 표시하거나 공유하면 안 되는지를 중심으로 정리합니다.

## 2. 전체 운영 흐름

1. 고객이 Cafe24 상품상세에서 인쇄용 파일을 업로드합니다.
2. 파일은 Naver Object Storage에 저장됩니다.
3. 파일 정보는 Supabase `files` 테이블에 저장됩니다.
4. 업로드 후 생성된 `file_id`가 Cafe24 주문 옵션의 업로드 파일 ID에 자동 입력됩니다.
5. 고객이 주문을 완료하면 Cafe24 Webhook 또는 Cafe24 주문 조회로 주문번호와 파일을 연결합니다.
6. 관리자는 `/admin`에서 오늘 처리할 파일을 확인합니다.
7. 필요한 파일을 다운로드해 검수합니다.
8. 검수 결과에 따라 관리자가 직접 파일 상태를 변경합니다.
9. 파일에 문제가 있으면 재업로드 요청 안내문 또는 재업로드 링크를 생성합니다.
10. 고객이 `/reupload?token=...` 링크에서 수정 파일을 다시 업로드합니다.
11. 재업로드 완료 파일은 새 `files` row로 저장되고, 재업로드 요청 이력의 새 파일 ID에 연결됩니다.
12. 관리자는 새 파일을 직접 다운로드해 확인한 뒤 상태를 변경합니다.
13. 기존 파일을 교체 처리해야 하면 관리자가 직접 기존 파일 상태를 `replaced`로 변경합니다.

## 3. /admin 접속 및 탭 구조

1. 운영 브라우저에서 `/admin`에 접속합니다.
2. 관리자 비밀번호로 로그인합니다.
3. 로그인 후 상단 탭에서 필요한 업무 화면으로 이동합니다.

현재 `/admin`은 업무 흐름 기준 5개 탭으로 구성되어 있습니다.

| 탭 | 용도 |
|---|---|
| 오늘 처리 | 매일 먼저 확인할 파일과 주의 로그를 보는 화면 |
| 파일 찾기 | 주문번호 또는 `file_id`로 파일을 찾고 상세 처리하는 화면 |
| 재업로드 | 고객에게 재업로드를 요청하고 새 파일을 확인하는 화면 |
| 로그 확인 | Webhook, 다운로드, 교정확인 이력을 확인하는 화면 |
| 설정 | Cafe24, Supabase, Naver Object Storage 연동 상태를 확인하는 화면 |

## 4. 오늘 처리 탭 사용법

오늘 처리 탭은 매일 업무 시작 시 가장 먼저 보는 화면입니다.

확인할 항목은 다음과 같습니다.

1. 최근 업로드 파일
2. 확인 전 파일
3. 재업로드 완료 파일
4. 확인 필요 Webhook 요약

### 최근 업로드 파일

최근 업로드 파일 목록은 빠르게 훑어보기 위한 요약형 목록입니다.

기본 표시 항목은 다음과 같습니다.

| 항목 | 설명 |
|---|---|
| 파일명 | 고객이 업로드한 원본 파일명 |
| `file_id` | 파일을 식별하는 고유 ID |
| Cafe24 주문번호 | 연결된 주문번호. 없으면 미연결 상태 |
| 상태 | 현재 파일 처리 상태 |
| 업로드일 | 파일 업로드 시각 |
| 다운로드 | 파일을 내려받는 버튼 |
| 상세 보기 | 파일 상세 화면으로 이동하는 버튼 |

오늘 처리 탭에서는 상태 변경 폼을 기본으로 펼쳐 보여주지 않습니다.

상태 변경이 필요하면 해당 행의 `상세 보기`를 눌러 `/admin?tab=files&file_id=<file_id>` 화면으로 이동한 뒤 처리합니다.

## 5. 파일 찾기 탭 사용법

파일 찾기 탭은 주문번호나 `file_id`로 파일을 찾고 실제 처리를 진행하는 화면입니다.

주요 기능은 다음과 같습니다.

| 기능 | 사용 방법 |
|---|---|
| Cafe24 주문 조회 테스트 | Cafe24 주문번호를 입력해 주문 상세와 주문 품목의 업로드 파일 ID를 확인합니다. |
| 주문번호로 파일 찾기 | Cafe24 주문번호를 입력해 연결된 파일 목록을 확인합니다. |
| `file_id`로 파일 찾기 | 업로드 파일 ID를 입력해 파일 상세 정보를 확인합니다. |
| 파일 상세 정보 | 파일명, 주문번호, 상태, 생성일 등 주요 정보를 확인합니다. |
| Cafe24 주문 정보 | 연결된 주문의 상품, 옵션, 업로드 파일 ID, 품목 수량, 품목 상태를 확인합니다. |
| 파일 다운로드 | 다운로드 버튼을 눌러 파일을 내려받습니다. |
| 주문번호 연결 | 필요한 경우 `file_id` 기준으로 주문번호를 연결합니다. |
| 파일 상태 변경 | 검수 결과에 따라 상태를 직접 변경합니다. |
| 주문번호 연결 이력 | 자동, 수동, 반자동 연결 이력을 확인합니다. |
| 상태 변경 이력 | 상태 변경 내역과 메모를 확인합니다. |
| 파일별 다운로드 로그 | 해당 파일의 다운로드 기록을 확인합니다. |

### Cafe24 주문 정보 섹션

파일 상세 화면에는 `files.order_id`가 연결된 경우 `Cafe24 주문 정보` 섹션이 표시됩니다.

이 섹션은 고객 개인정보 없이 주문, 상품, 옵션 중심으로 표시합니다.

현재 확인할 수 있는 항목은 다음과 같습니다.

| 항목 | 설명 |
|---|---|
| Cafe24 주문번호 | 파일과 연결된 주문번호 |
| `order_no` | Cafe24 응답에 있는 경우 표시되는 주문 보조 번호 |
| 주문일 | Cafe24 주문 생성 또는 결제 관련 시각 |
| 주문상태 | Cafe24 주문 상태 |
| 주문 품목 수 | Cafe24 주문에서 확인된 품목 수 |
| 품목 조회 상태 | 주문 품목 API 조회 성공, 실패, 미시도 여부 |
| 업로드 파일 ID 발견 수 | 주문 품목 옵션에서 찾은 업로드 파일 ID 개수 |
| 상품명 | 주문 품목의 상품명 |
| 상품번호 | Cafe24 상품번호 |
| variant code | 품목 옵션 코드 |
| 품목 수량 | `orderItemResponse.items[*].quantity` 기준 수량 |
| 품목 상태 | `orderItemResponse.items[*].order_status` 기준 상태값 |
| 상품 옵션 | Cafe24 품목 옵션 요약 |
| 추가 입력 옵션 | 업로드 파일 ID 등 추가 입력 옵션 요약 |
| 업로드 파일 ID | Cafe24 주문 품목에서 확인된 `file_id` |

품목 수량은 제작 수량을 확인할 때 참고합니다.

품목 상태는 Cafe24 품목 상태 코드값을 기준으로 표시합니다. 공식 문서 기준으로 확인된 `N00`은 `입금전 (N00)`처럼 한글 라벨과 원본 코드값을 함께 표시합니다. 아직 확인하지 않은 코드는 원본 코드값만 표시합니다.

### Cafe24 주문 정보 섹션에서 표시하지 않는 항목

다음 정보는 현재 `/admin` 파일 상세 화면에 표시하지 않습니다.

- 주문자명
- 수령자명
- 전화번호
- 이메일
- 주소
- 배송 메시지
- 주문 요청사항
- 결제 상세
- Cafe24 Admin API raw 응답 전체

`order.billing_name`과 `order.member_id`는 실제 존재가 확인되었지만 개인정보 또는 식별정보 가능성이 있어 현재 화면에는 표시하지 않습니다. 향후 표시하더라도 마스킹하거나 파일 상세 화면에 한정해야 합니다.

### 주문번호 연결 주의사항

- `files.order_id`가 비어 있으면 주문번호를 연결할 수 있습니다.
- 이미 주문번호가 연결되어 있으면 실제 Cafe24 주문상세와 `file_id`가 맞는지 다시 확인합니다.
- 수동 연결로 주문번호를 교체할 수 있는 경우, 현재 연결된 주문번호와 새 주문번호를 반드시 비교한 뒤 진행합니다.
- Webhook 자동 연결은 다른 주문번호가 이미 있으면 덮어쓰지 않고 `conflict_order_id`로 처리합니다.
- 잘못 연결한 주문번호는 다운로드, 검수, 재업로드 안내에 영향을 줄 수 있으므로 신중히 확인합니다.

## 6. 고객정보 필드 확인 결과

최근 관리자 전용 임시 확인 route로 Cafe24 주문 응답의 필드 존재 여부를 안전하게 확인했습니다.

확인 route는 결과 확인 후 삭제되었고, 현재 운영 코드에는 남아 있지 않습니다.

확인 결과는 `docs/ADMIN_CUSTOMER_INFO_FIELD_PRESENCE_RESULT.md`에 정리되어 있습니다.

### present 확인 필드

| 필드 | 상태 | 표시 기준 |
|---|---|---|
| `order.billing_name` | present | 이름으로 보일 수 있어 현재 미표시. 향후 마스킹 후 검토 |
| `order.member_id` | present | 회원 식별정보 가능성이 있어 현재 미표시. 향후 내부 확인용 또는 마스킹 검토 |
| `orderItemResponse.items[*].quantity` | present | 개인정보가 아니므로 품목 수량으로 표시 가능 |
| `orderItemResponse.items[*].order_status` | present | 개인정보가 아니므로 품목 상태로 표시 가능 |

### 기본 미표시 유지 항목

- 배송지 주소 전체
- 배송 메시지 전체
- 주문 요청사항 전체
- 결제 상세
- Cafe24 Admin API raw 응답 전체

## 7. 재업로드 탭 사용법

재업로드 탭은 파일에 문제가 있어 고객에게 수정 파일을 다시 받아야 할 때 사용합니다.

재업로드가 필요한 예시는 다음과 같습니다.

- 인쇄용 원본이 누락된 경우
- 해상도가 낮은 경우
- 칼선, 여백, 재단 위치 확인이 필요한 경우
- 주문 내용과 파일 내용이 다른 경우
- 고객에게 수정본 업로드를 요청해야 하는 경우

### 재업로드 요청 순서

1. 재업로드가 필요한 파일의 `file_id`를 확인합니다.
2. 재업로드 탭에서 해당 `file_id`를 검색합니다.
3. 재업로드 요청 안내문을 생성합니다.
4. 필요하면 재업로드 링크를 생성합니다.
5. 고객에게 안내문과 재업로드 링크를 전달합니다.
6. 고객은 `/reupload?token=...` 페이지에서 수정 파일 1개를 업로드합니다.
7. 재업로드 완료 후 새 파일 ID를 확인합니다.
8. 새 파일 상세 보기로 이동해 파일을 다운로드하고 검수합니다.
9. 검수 결과에 따라 새 파일 상태를 직접 변경합니다.
10. 기존 파일을 교체 처리해야 하면 관리자가 직접 기존 파일 상태를 `replaced`로 변경합니다.

### 재업로드 운영 원칙

- 재업로드 링크는 1회성 링크로 봅니다.
- raw token은 DB에 저장하지 않고 `token_hash`만 저장합니다.
- 고객이 재업로드를 완료하면 같은 링크로 다시 업로드할 수 없어야 합니다.
- 재업로드 파일은 새 `files` row로 저장됩니다.
- 재업로드 파일의 `files.customer_type`은 `cafe24-reupload`입니다.
- 기존 주문번호는 유지됩니다.
- `file_reupload_requests.new_file_id`에 새 파일 ID가 연결됩니다.
- 재업로드 요청 상태는 업로드 완료 후 `uploaded`로 처리됩니다.
- 재업로드 완료 후 기존 파일은 자동 삭제하지 않습니다.
- 재업로드 완료 후 기존 파일 상태를 자동으로 `replaced`로 바꾸지 않습니다.
- 재업로드 완료 후 새 파일 상태를 자동으로 `approved`로 바꾸지 않습니다.
- 새 파일 기본 상태는 `uploaded_pending`입니다.
- 관리자가 직접 새 파일을 확인한 뒤 `reviewing` 또는 `approved`로 변경합니다.

## 8. 로그 확인 탭 사용법

로그 확인 탭은 운영 기록과 문제 원인을 확인하는 화면입니다.

주요 기능은 다음과 같습니다.

| 기능 | 설명 |
|---|---|
| Cafe24 Webhook 수신 로그 | Cafe24에서 들어온 Webhook 처리 결과를 확인합니다. |
| Webhook processed_status 필터 | 정상, 주의, 실패 상태별로 로그를 걸러 봅니다. |
| 전체 다운로드 로그 | 관리자가 다운로드한 이력을 확인합니다. |
| 다운로드 로그 필터/CSV | `file_id`, 주문번호, 결과, 날짜 기준으로 필터링하고 CSV로 내려받습니다. |
| 전체 교정확인 이력 | 교정확인 요청과 고객 회신 기록을 확인합니다. |
| 교정확인 필터/CSV | 상태, `file_id`, 주문번호, 날짜 기준으로 필터링하고 CSV로 내려받습니다. |

### Webhook 상태값

정상 상태는 다음과 같습니다.

| 상태 | 의미 | 운영 판단 |
|---|---|---|
| `auto_linked` | 자동 연결 완료 | 정상 처리 |
| `already_linked` | 이미 연결됨 | 중복 수신이지만 정상 범위 |

확인이 필요한 상태는 다음과 같습니다.

| 상태 | 의미 | 확인할 것 |
|---|---|---|
| `received` | 수신만 된 상태 | 예전 로그인지, 자동 처리 전 로그인지 확인 |
| `no_order_id` | 주문번호 없음 | Webhook payload에서 주문번호를 찾지 못한 상태 |
| `no_file_id` | 업로드 파일 ID 없음 | Cafe24 주문 품목 옵션에 `file_id`가 있는지 확인 |
| `file_not_found` | 업로드 파일 없음 | Supabase `files`에 해당 `file_id`가 있는지 확인 |
| `conflict_order_id` | 다른 주문번호 연결됨 | 자동 덮어쓰기 금지. 주문상세와 `file_id`를 확인 |
| `failed` | 처리 실패 | API, 환경변수, 권한, DB 오류 가능성 확인 |

운영 원칙은 다음과 같습니다.

- `files.order_id`가 비어 있을 때만 자동 연결합니다.
- 이미 같은 주문번호가 연결되어 있으면 `already_linked`로 처리합니다.
- 다른 주문번호가 이미 있으면 덮어쓰지 않고 `conflict_order_id`로 처리합니다.
- Webhook raw payload 전체를 화면에 표시하지 않습니다.

## 9. 설정 탭 사용법

설정 탭은 앱 연동 상태를 확인하는 화면입니다.

확인 항목은 다음과 같습니다.

| 항목 | 확인 내용 |
|---|---|
| Cafe24 configured | Cafe24 API 설정이 준비되어 있는지 확인 |
| Supabase configured | Supabase 연결 설정이 준비되어 있는지 확인 |
| Naver Object Storage configured | 파일 저장소 설정이 준비되어 있는지 확인 |
| OAuth connection status | Cafe24 OAuth 연결 상태 확인 |
| access token 만료 시각 | API 연결 갱신 필요 여부 확인 |
| refresh token 만료 시각 | 장기 연결 갱신 필요 여부 확인 |
| scope 목록 | 앱에 부여된 Cafe24 권한 범위 확인 |

설정 탭은 상태 확인용입니다. token, secret, key 같은 민감정보 원문은 화면이나 문서에 적지 않습니다.

## 10. 고객용 파일 상태 조회 흐름

고객용 `/file-status` 페이지는 구현되어 있습니다.

고객은 주문번호와 업로드 파일 ID를 함께 입력해 파일 상태를 조회합니다.

운영 기준은 다음과 같습니다.

- `file_id` 단독 조회는 허용하지 않습니다.
- 주문번호 + `file_id` 조합이 맞아야 조회됩니다.
- Cafe24 품목별 주문번호 끝의 `-01`, `-02` 입력도 허용합니다.
- 예를 들어 상단 주문번호 `20260704-0000014`와 품목별 주문번호 `20260704-0000014-01` 모두 조회 대상이 될 수 있습니다.
- 조회 실패 시 어떤 값이 틀렸는지 구분하지 않는 통합 실패 문구를 사용합니다.
- 고객 화면에는 다운로드 버튼을 제공하지 않습니다.
- 고객 화면과 API 응답에는 `storage_path`, signed URL, `token_hash`, raw token, Webhook raw payload를 노출하지 않습니다.
- 모바일 입력칸 가독성 개선이 완료되어 긴 `file_id`도 입력하기 쉽게 정리되어 있습니다.

## 11. Cafe24 고객 화면 연결 상태

### Cafe24 고객 마이페이지 주문조회 > 주문상세 화면

Cafe24 고객 마이페이지 주문조회 > 주문상세 화면에는 파일 상태조회 안내 박스가 적용되어 있습니다.

운영 기준은 다음과 같습니다.

- 업로드 파일 ID가 보이는 옵션 영역 아래에 안내 박스를 둡니다.
- 주문처리상태 영역 위에 배치합니다.
- 버튼 문구는 `업로드 파일 상태 조회하기`입니다.
- 버튼은 `/file-status`로 이동합니다.
- URL에 주문번호나 `file_id`를 자동으로 붙이지 않습니다.
- 고객이 `/file-status`에서 주문번호와 업로드 파일 ID를 직접 입력합니다.
- PC와 모바일에서 확인이 완료되었습니다.

상품상세와 장바구니는 파일 상태조회 위치가 아닙니다. 상품상세에는 파일 업로드 위젯만 두고, 파일 상태조회는 주문 완료 후 고객이 다시 확인하는 주문조회/주문상세 또는 마이쇼핑 쪽에 둡니다.

### Cafe24 마이쇼핑 메인

Cafe24 고객 마이쇼핑 메인에는 `업로드 파일 상태 조회` 바로가기 카드가 적용되어 있습니다.

운영 기준은 다음과 같습니다.

- 주문조회 카드 바로 다음에 보조 카드로 둡니다.
- 보조 문구는 주문번호와 업로드 파일 ID가 필요하다는 취지입니다.
- 버튼은 `/file-status`로 이동합니다.
- URL 파라미터 자동 입력이나 Cafe24 화면 내 `file_id` 자동 파싱은 1차에서 하지 않습니다.
- 기존 주문조회, 회원정보, 배송주소록 등 주요 메뉴는 유지합니다.
- 기존 메뉴 숨김은 없습니다.

## 12. 파일 상태값 설명

| 상태값 | 화면 의미 | 언제 쓰는지 | 주의할 점 |
|---|---|---|---|
| `uploaded_pending` | 업로드됨 / 확인 전 | 고객이 파일을 업로드했고 아직 확인 전일 때 | 제작 진행 가능 상태로 보면 안 됩니다. |
| `reviewing` | 파일 확인 중 | 관리자가 파일을 열어 검수 중일 때 | 검수 중인 파일임을 내부에서 구분합니다. |
| `approved` | 파일 확인 완료 | 파일 검수가 끝났고 제작 진행이 가능한 상태로 판단될 때 | 자동 변경하지 않고 관리자가 직접 변경합니다. |
| `need_reupload` | 재업로드 요청 | 파일에 문제가 있어 고객 수정본이 필요할 때 | 재업로드 안내 또는 링크 생성과 함께 사용합니다. |
| `replaced` | 새 파일로 교체됨 | 새 파일이 들어와 기존 파일을 이전 파일로 남길 때 | 기존 파일은 삭제하지 않고 이력으로 보존합니다. |
| `archived` | 보관 처리 | 운영상 더 이상 처리하지 않는 파일을 보관할 때 | 다운로드와 이력 보존 목적을 고려해 신중히 사용합니다. |

## 13. 재업로드 상태값 설명

| 상태값 | 화면 의미 | 운영 기준 |
|---|---|---|
| `requested` | 재업로드 요청 | 고객에게 재업로드 링크를 보낸 상태 |
| `uploaded` | 재업로드 완료 | 고객이 새 파일을 업로드한 상태 |
| `reviewing` | 확인 중 | 관리자가 새 파일을 확인 중인 상태 |
| `completed` | 처리 완료 | 재업로드 후속 처리가 끝난 상태 |
| `expired` | 만료됨 | 링크 유효기간이 지난 상태 |
| `canceled` | 취소됨 | 재업로드 요청을 더 이상 사용하지 않는 상태 |
| `failed` | 실패 | 업로드 또는 처리 중 오류가 발생한 상태 |

## 14. 매일 업무 처리 순서

1. `/admin`에 로그인합니다.
2. 오늘 처리 탭을 엽니다.
3. 최근 업로드 파일을 확인합니다.
4. 확인 전 파일을 다운로드합니다.
5. 파일 찾기 탭의 상세 보기로 이동합니다.
6. Cafe24 주문 정보에서 상품명, 옵션, 품목 수량, 품목 상태를 확인합니다.
7. 파일을 검수합니다.
8. 정상 파일이면 상태를 `reviewing` 또는 `approved`로 변경합니다.
9. 문제가 있으면 상태를 `need_reupload`로 변경합니다.
10. 재업로드 요청 안내문을 생성합니다.
11. 필요하면 재업로드 링크를 생성해 고객에게 전달합니다.
12. 재업로드 완료 파일이 있으면 새 파일 ID를 확인합니다.
13. 새 파일을 다운로드해 검수합니다.
14. 기존 파일 교체 처리가 필요하면 관리자가 직접 기존 파일 상태를 `replaced`로 변경합니다.
15. 확인 필요 Webhook 로그가 있는지 확인합니다.
16. 다운로드 로그와 교정확인 이력을 확인합니다.

## 15. 문제 발생 시 확인 순서

일반적인 주문번호 또는 파일 연결 문제는 아래 순서로 확인합니다.

1. `/admin`에서 주문번호 검색이 되는지 확인합니다.
2. Cafe24 관리자 주문상세에 업로드 파일 ID가 들어갔는지 확인합니다.
3. `/admin` Cafe24 주문 조회 테스트로 주문번호를 조회합니다.
4. Supabase `files`에 해당 `file_id`가 있는지 확인합니다.
5. `cafe24_webhook_events`의 `processed_status`를 확인합니다.
6. Vercel Logs에서 `/api/cafe24/webhooks` 오류를 확인합니다.
7. 수동 연결 또는 Cafe24 주문 조회 기반 반자동 연결로 복구 가능한지 확인합니다.
8. 그래도 해결되지 않으면 Codex 수정 프롬프트를 작성합니다.

`/file-status` 조회 문제는 아래 순서로 확인합니다.

1. 입력한 주문번호가 상단 주문번호인지, 품목별 주문번호인지 확인합니다.
2. `-01`, `-02`가 붙어도 조회되는지 확인합니다.
3. `file_id`가 실제 Cafe24 주문 옵션에 있는지 확인합니다.
4. `/admin`에서 해당 `file_id`를 검색합니다.
5. `files.order_id`가 연결되어 있는지 확인합니다.
6. 실패 문구가 어떤 값이 틀렸는지 구분하지 않는지 확인합니다.

## 17. 절대 하지 말아야 할 것

- 승인 상태 `approved`는 자동으로 바꾸지 않습니다.
- 파일 검수는 관리자가 직접 합니다.
- `files.order_id`가 비어 있을 때만 자동 연결합니다.
- 이미 같은 주문번호가 연결되어 있으면 `already_linked`로 처리합니다.
- 다른 주문번호가 이미 연결되어 있으면 절대 덮어쓰지 않고 `conflict_order_id`로 처리합니다.
- 재업로드 완료 후 기존 파일은 자동 삭제하지 않습니다.
- 재업로드 완료 후 기존 파일 상태를 자동으로 `replaced`로 변경하지 않습니다.
- 새 파일 상태를 자동으로 `approved`로 변경하지 않습니다.
- 삭제 기능은 신중하게 접근합니다.
- 다운로드 로그와 상태 변경 이력은 보존합니다.

## 18. 운영자가 마지막으로 확인할 것

- 오늘 처리 탭에서 최근 업로드 파일이 요약형 목록으로 보이는지 확인합니다.
- 파일 찾기 탭에서 상세 보기, 다운로드, 상태 변경이 정상인지 확인합니다.
- 파일 상세의 Cafe24 주문 정보 섹션에서 품목 수량과 품목 상태가 보이는지 확인합니다.
- 파일 상세의 Cafe24 주문 정보 섹션에서 주문자명, 전화번호, 이메일, 주소가 보이지 않는지 확인합니다.
- 재업로드 탭에서 새 파일 ID, 새 파일 상세 보기, 새 파일 다운로드가 보이는지 확인합니다.
- 로그 확인 탭에서 Webhook 상태 필터, 다운로드 CSV, 교정확인 CSV가 보이는지 확인합니다.
- 설정 탭에서 연동 상태만 확인하고 민감정보 원문을 외부에 공유하지 않습니다.
